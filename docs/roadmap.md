# dotden — roadmap (post-v1)

Direction beyond v1. These are committed-ish intentions, not scoped work; the authoritative v1 boundary is [`scope-v1.md`](scope-v1.md).

- **Onboarding tour** that surfaces features (Workspaces, Scope, history, auto-sync) for new users, if needed.
- **Adopt existing/foreign chezmoi repos ("chezmoi integration").** Connect to a hand-crafted chezmoi repo without breaking it: add `.dotden/` non-invasively, and preserve features we don't expose (full templates, `run_` scripts, externals, gpg encryption) as read-only "advanced — managed via chezmoi CLI". v1 is greenfield-only; this is the planned bridge for power-user early adopters.
- **Visual block editor for per-OS content.** Keep a single File but mark sections as OS-specific blocks ("PATH on Windows" vs "PATH on Ubuntu") in an in-app editor. On Apply, an environment renders only its matching blocks plus the global ones, ignoring the others. This is a friendly GUI over chezmoi templates — the natural successor to OS **Scope**.
- **Soft-block secrets + in-flow PM migration — _delivered in v1_, no longer roadmap.** Once a v1.5 refinement; now the shipped v1 design: commit-time soft-warn (never block), explicit Commit-anyway bypass, at-detection Convert to a Secret reference (1Password / Bitwarden / `pass`) via the two-step `SecretWarning` → `SecretPicker` flow. Mapped in [`user-flow/journeys/05-secrets.md`](user-flow/journeys/05-secrets.md); onboarding hard-block → soft-warn already reconciled in `design-system/screens/secret-and-errors.md`. Kept here only as a pointer — the remaining secrets roadmap work is age encryption (below).
- **Encryption (age)** for users who want secrets in the repo without a password manager (requires out-of-band key distribution to each environment). _The genuine post-v1 secrets item._
- **Provider convenience/automation layer (committed direction, per-Provider, GitHub-first; see [ADR 0020](adr/0020-provider-agnostic-pure-git-floor-v1-lean-auth.md)).** The friction-reducing layer above the v1 pure-git floor: OAuth/device-flow sign-in (so the user never sets up git creds or pastes a PAT), one-click Remote creation (the old "Create Den"), and API-based change detection. Built one Provider at a time — GitHub, then GitLab, Bitbucket, self-hosted GitLab/Gitea. **Bare git (arbitrary SSH URL + PAT/SSH) stays the universal escape hatch**, supported by the floor with no conveniences. Open sub-decision: per-Provider-optimal auth (GitHub App for least-privilege on GitHub) vs one-portable-OAuth-everywhere.
- **Manual edit of the merged buffer in the Conflict resolver (planned).** v1 ships per-hunk **Keep / Take / Both** (deterministic, covers the common collisions); later, a free-form **hand-edit of the resolved buffer in-center** for the cases order-alone can't fix (scope-v1's "Open-both", fully realized). No new merge engine — the `ConflictModel` already owns resolved-bytes (ADR 0008); this just lets a human author those bytes directly. Reconcile [`user-flow/journeys/04-conflicts.md`](user-flow/journeys/04-conflicts.md) + the conflict-resolver screen spec when it lands.
- **3-way drift merge — the apply-time "Resolve…" option (planned).** When Apply hits a File with **uncommitted local edits**, v1 **blocks-with-warning** and offers Commit-first / Discard (journey 04, ADR 0038). Later, a third option — a **3-way merge** (destination · source · target) surfaced in the same guard, the dotden-owned equivalent of `chezmoi merge` (which we don't call — it's an interactive vimdiff tool, ADR 0038). Built on the **same axis-agnostic `ConflictModel`** as the cross-env resolver, so it's additive, not a second engine. The v1 block-and-warn is the reserved socket. Reconcile [`user-flow/journeys/04-conflicts.md`](user-flow/journeys/04-conflicts.md) when it lands.
- **Optional note on a conflict-resolution history entry (planned).** v1 records a resolved cross-env Conflict as a merge commit with an **auto-generated message** (e.g. *"Merge work-laptop into macbook — resolved ~/.zshrc"*), visible in File history, no user input required. Later, let the user attach an **optional note** ("why I took theirs") to that entry — frictionless by default, annotatable when it matters. Reconcile [`user-flow/journeys/04-conflicts.md`](user-flow/journeys/04-conflicts.md) when it lands.
- **Always-on background daemon** beyond the tray.
- **MCP server inside the app.** Ship an opt-in MCP server users enable from within dotden, letting an LLM connect to the running app and interact with the user's Den — query/track/apply Files, inspect Workspaces/Scope, etc. Same model as Figma's in-app MCP (LLM talks to the installed app, not a cloud API). Rough idea only, not scoped.
